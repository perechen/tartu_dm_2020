Genre classification with SVM

Here we show that vectors of word frequencies hold more information than a simple authorship. Using roughly the same methods and features we can tackle the problem of genre classification, restating our questions on similarities and differences on the meta-level of genre instead of an author.

Corpus

We use 18th century French drama for this experiment that is labeled as either Comedy or Tragedy.

  • Data come from open-source digital collection Théâtre Classique.
  • 249 dramatic texts which had words “comédie” or “tragédie” in the genre definitition (not everything that was availiable made it’s way in our corpus: texts too short were trimmed out)

Classic French drama comes from a literary system that was controlled by rigid normative aesthetics, distinct public expectations and even space itself (genres were differently represented in different theaters). 17-18th century French theater was an icon of play stratification across genres for the whole Europe. Obviously, we expect very high accuracy in distinguishing between Comedies & Tragedies - but would a simple word frequencies be enough?

Method 1: classify() with “stylo” package

{stylo} package has a classify() function with GUI for supervised classifications. Several algorithms are available (including classification by nearest neighbor), but we’re using SVM here.

NB!!! Function looks in “primary_set” folder for training set and “secondary_set” for test set. Make sure you have these folders in your working directory. These folders should be also filled with data (which maybe not very convenient)

library(stylo)

svm_results=classify(gui = F, # you can obviously run plain classify() function to set everything in GUI
         method="svm",
         kernel="linear",
         cost=1)

Method 2. svm() from “e1071”

Now when we have represented texts by feature vectors, we can start splitting our data to test and train sets. Before that, however, let’s check how two genres are represented in our corpus

library(e1071)

z_fin$text_genre %>% table()
## .
##   C   T 
## 154  95

We have a disparity here: comedies are over-represented. It’s generally a good idea to keep data points per class (genre) equal, so algorithm would have equal possibilities for learning features of each class. Let’s downsample comedies making them not more frequent than tragedies. After downsampling we can split data to training and test: let’s make it with a proportion of 60/40 - a bit funnier than usual 75/25 used everywhere in machine learning.

set.seed(789) # seed for reproducibility

n = min(table(z_fin$text_genre)) # what is the value of the less frequent class

## in what proportion split to training and test sets (roughly 60% of texts will go to training)
train_size = round(n*0.60)
test_size = n - train_size


train_set = z_fin %>%
  group_by(text_genre) %>%
  sample_n(train_size) # sample n times per each group (genre) 

test_set = z_fin %>% 
  anti_join(train_set, by="text_id") %>% # 1. remove already sampled training set from the data
  group_by(text_genre) %>% 
  sample_n(test_size)  # 2. sample again the test per each genre

Now we can fit SVM model to our training data (learn genres). We are using linear kernel, and relatively low cost value (it determines how hard margin violations are punished)

svm_model <-svm(as.factor(text_genre)~.,  # we try to predict genre with all words (GENRE ~ WORDS), so we take all columns as predictor variables, we use . to select all data
                data=train_set %>% select(-text_id), # fit on train data, remove ids, 
            method="C-classification", 
            kernel="linear", 
            cost=1,
            scale=T)

Feature importance

summary() gives us some info on the fitted model, but the variable svm_model holds even more of that (try, for example to access this variable through $ sign).

Using the information from Support Vectors and their relations to features + the coefficient that give us an idea of the hyperplane slope we can compute feature weights. This gives us a nice insight to model performance and feature contribution to class distinctiveness.

### compute features weights (multiply coefficients by support vectors distances to features)
w = t(svm_model$coefs) %*% svm_model$SV
moins cet ou jamais voilà seigneur dit ciel veux ont voir yeux fils toujours êtes
-0.011 -0.01 0.019 0.027 0.01 -0.034 0.007 -0.017 0.042 -0.032 0.03 -0.011 -0.009 -0.009 -0.012

So, which words contribute most to differences in Tragedy and Comedy according to fitted SVM? We can interpret weights that are higher than zero as important for class “comedy”, while lower than zero weights would associate with “tragedy”.

Hey, but how do we know that? We are doing binary classification (on two classes). Because the reference (first) class in our summary is “C” - which is comedy, so positive values associate with importance for this genre. Negative values, therefore, negatively associate with Comedy… which means they associate with Tragedy (remember, a binary classification!).

This means we can annotate features accordingly and display most important features for each class/genre and learn something of how SVM learned differences between Comedy & Tragedy.

tibble(weight=w[1,], word=colnames(w)) %>%  # construct a tibble from weights
  mutate(genre = case_when(weight > 0 ~ "Comedie", # if weight > 0 - put a Comedie label
                           weight < 0 ~ "Tragedie")) %>% # weight < 0 - put a Tragedie label
  group_by(genre) %>% 
  mutate(abs=abs(weight)) %>% # take absolute values
  top_n(20,abs) %>%  # take 20 most distinctive features per each genre
  
  ggplot(aes(reorder(word,abs),abs,fill=genre)) + geom_col() + # plot
  coord_flip() + 
  facet_wrap(~genre,scales="free")

Method 3. SVM with “tidymodels” workflow

Now, admittedly using svm() from {e1071} requires a bit of meddling and hacking. Recently a meta-package {tidymodels} for statistical and supervised started to gain popularity because how explicit it makes the modeling process and how easy it can be to setup tuning and tweaking models.
Roughly speaking, tidymodels provides an uniform way to build an instruction of a model and then execute it when fitting the model. Generally each “instruction” or “workflow” is build out of two modules:

  1. Data recipe. Setting up dependent variables and predictors, recording the steps that you take to prepare data for the model;
  2. Model specification. Description of the model you use, setting up it source (“engine”) and parameters.

A recipe and a model specs are then combined into one “workflow” - the whole instruction list, which can be later fitted, cross-validated, resampled, etc. Current tidymodels together with package textrecipes provide in-workflow way of dealing with textual data - so all the steps that we take manually (tokenization, filtering, counts), could be frist recorded in a reciepe and then executed when model is fitted.

It almost makes a model building with tidymodels an exercise in arranging LEGO blocks in the right order: it may bring benefits (clear structure, modularity, high control of the fitting), but also limitations (dependence on some third-party defined functions to do specific things, lack of preprocessing control, lack of model interpretability).

To learn more about predictive text modeling with tidymodels in R check Supervised Machine Learning for Text Analysis in R

We will need following libraries:

We can start assembling set of instructions for data preparations. It’s a recipe where we put steps that should be taken with the original data.

genres_recipe =
  recipe(genre~text,data=genre_train) %>% # genre~text formula means that we try to predict genre with text column
  step_tokenize(text) %>%  # now we define how to transform text to features. 1. tokenize
  step_tokenfilter(text, # 2. filter tokens. We use words that appear in the corpus at least 10 times
                   min_times = 10,
                   max_tokens = 1000) %>% # use first 1000 most frequent words
  step_tf(text, weight_scheme = "term frequency") %>%  # 3. we calculate relative frequencies of each word to account for different sizes of texts
  step_scale(all_predictors()) # 4. we scale frequencies. because at this points our "text" variable is transformed to matrix with word frequencies, we use "all_predictors()" to apply scaling

Next we define model specifications. We use SVM with polynomial kernel from kernlab package. Notice that degree is set to 1: 1-order polynomial is a line, so our kernel is linear. Generally we don’t want to use any kernel tricks with multidimensional text data - so a linear kernel is just fine!

svm_spec <- svm_poly(cost=1,degree = 1) %>% # we use polynomial kernel for SVM with degree of 1 - which means we use linear kernel
  set_mode("classification") %>%
  set_engine("kernlab")

Finally we can combine recipe and the model definition into the one “workflow” that keep record of all the things we do for modeling (see it’s output)

genre_svm_wf <- workflow() %>%
  add_recipe(genres_recipe) %>%
  add_model(svm_spec)

As in all supervised modeling our “test” set is very valuable “unseen” data that we can use only once. To fine-tune the model and test general behavior of our SVM we must rely on “training” set only. There is a well-established procedure of “cross validation” that is used to iteratively fit the model on different sub-training and sub-test sets - in a fashion that all available data is used in training and testing to exclude potential bias in arbitrary data splits.

So before we do the final fit on our test set, let’s use “V-fold cross-validation”. It will “fold” our training set in V number of chunks (we use 10). When we will perform iterative resampling, each of the folds will be used as the test set, while remaining 9 folds will be put to training. In this case all chunks will appear both as training and testing. Luckily with tidymodels you can fold data in one line:

genre_folds <- vfold_cv(genre_train,strata = "genre",v = 10)

)
